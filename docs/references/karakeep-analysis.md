# Karakeep Reference Analysis

## Source
- reference repo provided by the user: `https://github.com/karakeep-app/karakeep`

## What mattered for Minakeep
- Karakeep demonstrates a broad personal-knowledge-vault shape rather than a simple note blog.
- Its useful inspiration for Minakeep is the combination of private capture plus later retrieval, not the full feature inventory.
- Feature fronts that are clearly separable for Minakeep include notes, links, publishing, search/tagging, and AI enrichment.
- The reference also demonstrates a dense, premium UI that treats saved knowledge as primary product value instead of as secondary blog chrome.

## What Minakeep intentionally does not copy in v1
- collaboration and multi-user workspaces
- importer pipelines and automation rules
- OCR, crawler enrichment, and archived content snapshots
- mobile or extension-specific surfaces
- chat-style assistant UX

## Planning takeaway
- the reference is useful for product framing, but Minakeep should stay smaller and more explicit than Karakeep
- the next queue should stay feature-sliced instead of creating one broad “AI + redesign” task
- public publishing remains limited to notes even though the reference product is broader
- the UI refresh should borrow density, polish, and layout confidence from the reference while keeping Minakeep’s route model and brand distinct
